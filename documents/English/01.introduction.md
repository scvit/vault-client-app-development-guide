# Overview of Application Development Using Vault Secret

## Development Goals

**Core Principle: Never embed passwords or API keys in your code.**

*   ❌ **Legacy Method**: Hardcoding like `password = "mypassword123"` in the code.
*   ✅ **Modern Method**: Dynamically fetching secrets from Vault.

### Why is this necessary?

1.  **Enhanced Security**: Secrets are not exposed even if the code is leaked.
2.  **Centralized Management**: All secrets are managed in one place.
3.  **Automatic Renewal**: Secrets are automatically updated without code changes.
4.  **Access Control**: Fine-grained control over who can access which secrets.

## What is Vault?

**HashiCorp Vault** is a tool for securely storing and managing secrets (passwords, API keys, certificates, etc.).

*   **Secret Storage**: Encrypts and stores secrets like database passwords and API keys.
*   **Dynamic Secrets**: Generates temporary credentials on-demand (with automatic expiration).
*   **Access Control**: Fine-grained control over which applications can access which secrets.
*   **Audit Logs**: Records who accessed which secret and when.

## Considerations for Development

1.  Vault provides both a static Key/Value (KV) store and Secret Engines for dynamic credentials.
2.  For initial adoption, it is recommended to manage existing credentials with KV and then gradually migrate to using Secret Engines.
3.  Development should consider both the KV approach for initial implementation and the future transition to Secret Engines.

## Option 1: Direct Vault API Integration (Recommended ✅)

### What is this approach?

**The application communicates directly with the Vault server to fetch secrets.**

### Authentication Process (Simplified)

1.  **AppRole Authentication**: How an application logs into Vault.
    *   `role_id`: The application's identifier (public).
    *   `secret_id`: The application's password (private).

2.  **Token Issuance**: Upon successful login, Vault issues a temporary token.
    *   This token is used to access secrets.
    *   The token expires after a certain period for security.

3.  **Token Renewal**: The token must be renewed before it expires.
    *   Code for automatic renewal must be implemented.

### What Developers Need to Do

#### Step 1: Initial Setup by Vault Administrator (Developers can ignore this)

```bash
# Commands executed by the Vault administrator
vault auth enable approle
vault write auth/approle/role/myapp \
    secret_id_ttl=24h \
    token_ttl=1h \
    token_max_ttl=24h \
    period=1h \
    policies="myapp-policy"
```

**Provided to the developer**: `role_id` and `secret_id` (via environment variables or configuration files).

#### Step 2: Application Login to Vault

```http
POST /v1/auth/approle/login
{
  "role_id": "your_app_role_id",
  "secret_id": "your_app_secret_id"
}
```

**Response**: `client_token` (used to access secrets).

#### Step 3: Token Renewal (Important!)

```http
POST /v1/auth/token/renew-self
X-Vault-Token: <your_token>
```

**Why is this needed?** The token expires after 1 hour, so it must be renewed periodically for the application to continue running.

#### Step 4: Fetching a Secret

```http
GET /v1/kv-demo/data/sample
X-Vault-Token: <your_renewed_token>
```

**⚠️ Important**: When using the KV v2 secrets engine, the path must include `/data/`.
- CLI: `vault kv get kv-demo/sample`
- API: `/v1/kv-demo/data/sample`

**Example Response**:
```json
{
  "data": {
    "data": {
      "username": "dbuser",
      "password": "secretpassword123"
    }
  }
}
```

#### Step 5: Using the Secret in the Application

```javascript
// ❌ Do not do this
const password = "hardcoded_password";

// ✅ Do this instead
const password = vaultResponse.data.data.password;
// Use only in memory, do not save to a file.
```

### Required Configuration Information

Developers need to include the following in their configuration:

*   **Vault Server Address**: `https://vault.company.com:8200`
*   **Namespace**: The Vault area the application can access (usually segregated by team).
*   **Secret Path**: The path to the secret the application can access (e.g., `/kv/data/database`).
*   **Automatic Renewal**: Code to automatically renew the token before it expires is mandatory.

### Language-Specific Libraries (Developer Tools)

*   **C**: Use libcurl + json-c (see example).
    -   **Complete Example**: [C Vault Client](./c-app/) (for developer distribution).
    -   **Supported Features**: KV v2, Database Dynamic, Database Static secrets engines.
    -   **Advanced Features**: Real-time renewal, version-based caching, TTL-based renewal.
    -   **Security**: Entity-based permissions, automatic token renewal, memory security.
    -   **Configuration**: INI file-based configuration, multi-secrets engine support.

*   **Java**: Use Apache HttpClient + Jackson (see example).
    -   **Complete Example**: [Java Vault Client](./java-pure-app/) (for developer distribution).
    -   **Supported Features**: KV v2, Database Dynamic, Database Static secrets engines.
    -   **Advanced Features**: Real-time TTL calculation, version-based caching, multi-threaded renewal.
    -   **Security**: Entity-based permissions, automatic token renewal, memory security.
    -   **Configuration**: Properties file-based configuration, system property override support.
    -   **Compatibility**: Tested on Java 11+, Maven-based build.

**The pattern is the same for all languages**: Login → Renew Token → Fetch Secret.

## Option 2: Using Vault Proxy

### What is Vault Proxy?

**Vault Proxy** is an intermediary server that sits between the application and Vault.

*   **Application's Role**: Requests secrets from the Proxy.
*   **Proxy's Role**: Fetches secrets from Vault and delivers them to the application.
*   **Advantage**: Eliminates the need for complex authentication/token management code in the application.

### Differences from a Developer's Perspective

**Option 1 (Direct Integration)**:
```javascript
// Complex code required
const token = await vaultLogin(roleId, secretId);
const secret = await vaultGetSecret(token, path);
await vaultRenewToken(token); // Run periodically
```

**Using Vault Proxy**:
```javascript
// Very simple code
const secret = await fetch('http://vault-proxy:8200/v1/kv/data/db');
```

### Pros and Cons

**Pros**:
- Application code becomes very simple.
- No need for token management logic.
- Faster development speed.

**Cons**:
- Requires additional operation of a Proxy server.
- The Proxy holds all tokens, so its security design is critical.

## Option 3: Vault Agent + Environment Variable/File Injection (For Legacy Apps)

### When to use this?

Used when **API integration is difficult**:

*   **Legacy Apps**: Older applications where HTTP API integration is challenging.
*   **External Tools**: Third-party tools (DB clients, monitoring tools, etc.).

### How it works

1.  **Vault Agent** runs in the background.
2.  The Agent periodically fetches secrets from Vault.
3.  It injects secrets directly as **environment variables** (no file storage).
4.  The Agent executes a C program, passing the environment variables.
5.  The application reads the secret from the environment variables using `getenv()`.

### Differences from a Developer's Perspective

**Legacy Method (Hardcoding)**:
```c
// ❌ Do not do this
char* password = "hardcoded_password";
```

**Using Vault Agent**:
```c
// ✅ Read from environment variable
char* password = getenv("DB_PASSWORD");
```

**Reading from Environment Variables (Recommended)**:
```c
// ✅ Read directly from environment variables (injected by Agent)
char* password = getenv("FROM_VAULT_SECRET");
char* source = getenv("VAULT_SECRET_SOURCE");
char* timestamp = getenv("SECRET_TIMESTAMP");
```

**Or Reading from a Configuration File**:
```c
// ✅ Read from a file (automatically updated by Agent)
// config.ini content:
// [database]
// password = password_from_vault
```

### Example Configuration

**Vault Agent Configuration** (`vault-agent.hcl`):
```hcl
# AppRole authentication settings
auto_auth {
  method {
    type = "approle"
    namespace = "skt_demo"
    mount_path = "auth/approle"
    config = {
      role_id_file_path = "./role_id"
      secret_id_file_path = "./secret_id"
      remove_secret_id_file_after_reading = false
    }
  }
  
  sink "file" {
    config = {
      path = "./token"
    }
  }
}

# Environment variable template settings (injects directly, does not save to file)
env_template "FROM_VAULT_SECRET" {
  contents = "{{ with secret \"kv-demo/data/sample\" }}{{ .Data.data.sample_secret }}{{ end }}"
  error_on_missing_key = true
}

env_template "VAULT_SECRET_SOURCE" {
  contents = "vault-agent"
}

env_template "SECRET_TIMESTAMP" {
  contents = "{{ with secret \"kv-demo/data/sample\" }}{{ .Data.metadata.created_time }}{{ end }}"
  error_on_missing_key = true
}

# Execute C program (Agent runs it with environment variables)
exec {
  command                   = ["./run_c_app.sh"]
  restart_on_secret_changes = "always"
  restart_stop_signal       = "SIGKILL"
}
```

**Application Execution Script** (`run_c_app.sh`):
```bash
#!/bin/bash

# Terminate previous instance
pkill -f "env-reader"

# Wait for 1 second
sleep 1

# Execute C program (environment variables are automatically injected by Agent)
cd /path/to/c-app
./env-reader
```

**How to Run**:
```bash
# Run the Agent (injects env vars and runs the C program)
vault agent -config=vault-agent.hcl
```

## Comparison and Selection Guide

### Comparison Table

| Category | Option 1 (Direct API) | Option 2 (Proxy) | Option 3 (Agent) |
|---|---|---|---|
| **Dev Difficulty** | 🔴 Hard (Token Mgmt) | 🟢 Easy | 🟡 Medium |
| **Security** | 🟢 High (Dynamic Tokens) | 🟢 High | 🟡 Medium (File Exposure) |
| **Flexibility** | 🟢 High (All Features) | 🟢 High | 🟡 Limited |
| **Ops Complexity** | 🟡 Medium | 🔴 High (Proxy Mgmt) | 🟢 Low |
| **Suitable For** | New Apps | Simple Apps | Legacy Apps |

### Selection Guide

#### **New Application Development**
→ **Option 1 (Direct API Integration)** is recommended.
- Most secure and flexible.
- Requires implementation of token renewal logic.
- **C**: No need to implement token renewal logic by using the [complete example](./c-app/).
- **Java**: No need to implement token renewal logic by using the [complete example](./java-pure-app/).

#### **Simple Applications**
→ **Option 2 (Vault Proxy)**
- Fast development speed.
- Requires operation of a Proxy server.

#### **Legacy Applications**
→ **Option 3 (Vault Agent)**
- Minimizes changes to existing code.
- Only requires adding logic to read from environment variables/files.

### Final Recommendation

**Development Goal**: Never embed secrets in code.

**Recommended Order**:
1.  **New Development** → Option 1 (Direct API) - **Utilize complete examples for C/Java**.
2.  **Simple Apps** → Option 2 (Vault Proxy).
3.  **Legacy Apps** → Option 3 (Vault Agent).

**Common Principles**:
- Use secrets only in memory.
- Do not save secrets to files or logs.
- Consider token expiration times.
