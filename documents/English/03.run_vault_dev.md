# Vault Development Server Setup Guide

## 📋 Prerequisites

- Download and extract Vault binary for execution
  - https://www.vaultproject.io/downloads
- Docker environment for MySQL execution
- For SSH verification, a server with configured SSH Engine (OTP / Signed Certificate) is required.

For a batch application script that applies the following steps, refer to the `setup-vault-for-my-vault-app.sh` file.

### Step 1: Run Vault in Development Mode

Download Vault from <https://developer.hashicorp.com/vault/install> and extract it.

Execute the following command to run Vault:
```bash
# Run Vault in development mode (auto unseal, includes default policies)
vault server -dev -dev-root-token-id="root"
```

**Development Mode Features**:
- Automatically unsealed
- Default root token: `root`
- Data stored only in memory (initialized on restart)

### Step 2: Set Vault Environment Variables

Linux/Unix/MacOS:
```bash
export VAULT_ADDR='http://127.0.0.1:8200'
export VAULT_TOKEN='root'
```

Windows:
```batch
set VAULT_ADDR=http://127.0.0.1:8200
set VAULT_TOKEN=root
```

### Step 3: Enable AppRole Authentication

In another command window, execute the following command to enable AppRole authentication:

```bash
# Enable AppRole authentication method
vault auth enable approle
```

#### 🏗️ **Create Entity and Connect AppRole**

Entity is used as a unique identifier for the app. This identifier is connected to AppRole and used for Policy and Secret paths.

```bash
# Create Entity (unique identifier per app)
vault write identity/entity name="my-vault-app" policies="myapp-templated-policy"

# Check Entity ID information
vault read -field=id /identity/entity/name/my-vault-app

# Check AppRole Accessor ID information
vault read -field=accessor sys/auth/approle

# Get Role ID
vault read auth/approle/role/my-vault-app/role-id

# Create Entity Alias
vault write identity/entity-alias name="<role_id>" canonical_id="<entity_id>" mount_accessor="<approle_accessor_id>"

# Connect AppRole to Entity
vault write auth/approle/role/my-vault-app \
  secret_id_ttl=1h \
  token_ttl=0 \
  token_max_ttl=0 \
  period=1m \
  orphan=true

# Generate Secret ID
vault write -f auth/approle/role/my-vault-app/secret-id
```

- secret_id_ttl sets the validity period of Secret ID. Set to 1 hour for testing. Generally, this maintenance time is set to about the app boot time.
- period sets the token renewal cycle. Set to 1 minute for testing.

### Step 4: Advanced Policy Configuration Using Policy Templating

#### 🔐 **Create Entity-based Policy**

Using [Vault ACL Policy Templating](https://developer.hashicorp.com/vault/tutorials/policies/policy-templating), configure a policy that specifies a unique entity per app and allows KV paths with that name.

```bash
# Create Entity-based template policy
vault policy write myapp-templated-policy - <<EOF
# App-specific secret paths (based on identity.entity.name)
path "{{identity.entity.name}}-kv/data/*" {
  capabilities = ["read", "list"]
}

path "{{identity.entity.name}}-database/creds/*" {
  capabilities = ["read", "list"]
}

path "{{identity.entity.name}}-database/static-creds/*" {
  capabilities = ["read", "list"]
}

# Token renewal permission
path "auth/token/renew-self" {
  capabilities = ["update"]
}

# Token lookup permission
path "auth/token/lookup-self" {
  capabilities = ["read"]
}
EOF
```

### Step 5: Enable KV Secret Engine

```bash
# Enable KV v2 secret engine
vault secrets enable -path=my-vault-app-kv kv-v2

# Create example secret data
vault kv put my-vault-app-kv/sample \
  sample_secret="my-secret-value-123" \
  database_url="mysql://localhost:3306/mydb"
```

### Step 6: Enable Database Secret Engine

Using [Vault Database Secrets Engine](https://developer.hashicorp.com/vault/docs/secrets/databases), enable the Database secret engine. Here, we use MySQL running in Docker.

```bash
# Run MySQL container (use existing MySQL if available)
docker run --name my-vault-app-mysql -e MYSQL_ROOT_PASSWORD=password -d -p 3306:3306 mysql:9
```

```bash
# Create Static User with only password replacement
mysql -u root -ppassword -h 127.0.0.1 -P 3306 --protocol=TCP -e "CREATE USER 'my-vault-app-static'@'%' IDENTIFIED BY 'password'; GRANT ALL PRIVILEGES ON *.* TO 'my-vault-app-static'@'%';"
```

```bash
# Enable Database secret engine
vault secrets enable -path=my-vault-app-database database

# Configure Database secret engine
vault write my-vault-app-database/config/mysql-demo \
  plugin_name=mysql-database-plugin \
  connection_url="{{username}}:{{password}}@tcp(localhost:3306)/" \
  allowed_roles="*" \
  username="root" \
  password="password"

# Create Dynamic Role
vault write my-vault-app-database/roles/db-demo-dynamic \
  db_name=mysql-demo \
  creation_statements="CREATE USER '{{username}}'@'%' IDENTIFIED BY '{{password}}'; GRANT ALL PRIVILEGES ON *.* TO '{{username}}'@'%';" \
  default_ttl="1h" \
  max_ttl="24h"

# Test Dynamic Role (check validity period, Username, Password)
vault read my-vault-app-database/creds/db-demo-dynamic

# Create Static Role (set password replacement schedule)
vault write my-vault-app-database/static-roles/db-demo-static \
  db_name=mysql-demo \
  username=my-vault-app-static \
  rotation_schedule="0 * * * *"

# Test Static Role (check current password and remaining time)
vault read my-vault-app-database/static-creds/db-demo-static
```

**Advantages of Policy Templating**:
- **Dynamic Paths**: Automatically generate unique paths per app using `{{identity.entity.name}}` pattern
- **Security Isolation**: Each app can only access its own secrets
- **Operational Efficiency**: No need to modify policies when adding new apps

## 🔍 Configuration Verification

### 1. Check Vault Status

```bash
# Check Vault server status
vault status

# Check authentication methods
vault auth list

# Check secret engines
vault secrets list
```

### 2. Check Policies

```bash
# Check created policies
vault policy list

# Check policy content
vault policy read myapp-templated-policy
```

### 3. Check Entity and AppRole

```bash
# Check Entity list
vault list identity/entity

# Check AppRole configuration
vault read auth/approle/role/my-vault-app
```

### 4. Test Secret Access

#### 4.1. Direct Testing Using Vault CLI
```bash
# Get Role ID
vault read auth/approle/role/my-vault-app/role-id

# Generate Secret ID
vault write -f auth/approle/role/my-vault-app/secret-id

# Test AppRole login
vault write auth/approle/login role_id="<role_id>" secret_id="<secret_id>"

# Test secret retrieval
vault kv get my-vault-app-kv/sample
```

#### 4.2. Script Testing (Vault Proxy Required)
To test script examples, Vault Proxy is required:

```bash
# Set up and run Vault Proxy
cd samples/script-sample/vault-proxy-demo
./setup-token.sh
./start-proxy.sh

# Test scripts
cd ..
./get_kv_secret_proxy.sh
./get_db_dynamic_secret_proxy.sh
./get_db_static_secret_proxy.sh

# Stop Vault Proxy
cd vault-proxy-demo
./stop-proxy.sh
```

**Why is Vault Proxy needed?**
- Scripts use pure curl without Vault CLI
- Vault Proxy handles complex authentication processes (AppRole login, token renewal)
- Scripts can retrieve secrets with simple HTTP requests

## ⚠️ Precautions

### Development Environment Only
- Use this configuration **only in development environments**
- Production environments require appropriate security settings

### Data Persistence
- Development mode is memory-based, so all data disappears when the server restarts
- Use file-based storage backend for permanent storage if needed

### Security Considerations
- Never use `root` token in production environments
- Maintain appropriate token TTL settings
- Perform regular security audits
