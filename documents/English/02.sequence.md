# Vault Secret Integration Process

## 📋 Overview

This document describes the entire process for an application to securely fetch secrets by integrating with Vault. It uses sequence diagrams to represent a universally applicable procedure, regardless of the programming language.

## Option 1: Direct Vault API Integration (Recommended ✅)

1.  Vault Authentication and Token Issuance
2.  Token State Management
3.  Token Renewal Process
4.  Fetching Static Secrets (KV)
5.  Managing Dynamic Secrets (TTL-based)
6.  Integrated Process (Full Flow)

---

### 🔐 1.1: Vault Authentication and Token Issuance

#### AppRole Login Process

```mermaid
sequenceDiagram
    participant App as Application
    participant Vault as Vault Server

    Note over App,Vault: 1. Initial Authentication Process

    App->>Vault: POST /v1/auth/approle/login
    Note right of App: Send role_id, secret_id

    Vault-->>App: client_token + lease_duration
    Note left of Vault: Issue token and TTL info

    Note over App: Store token and calculate expiration
    Note over App: token_expiry = now + lease_duration
```

#### Token State Management

```mermaid
sequenceDiagram
    participant App as Application
    participant Timer as Timer/Scheduler

    Note over App,Timer: 2. Periodic Token Lifetime Check

    loop Token State Monitoring (every 10s)
        Timer->>App: Request to check token state
        App->>App: Calculate current_time - token_issue_time
        App->>App: Check if 4/5 of TTL is reached

        alt Token renewal needed (4/5 of TTL reached)
            App->>App: Start token renewal process
        else Token is OK
            App->>Timer: Token state OK
        end
    end
```

### 🔄 1.2: Token Renewal Process

#### Token Renewal and Failure Handling

```mermaid
sequenceDiagram
    participant App as Application
    participant Vault as Vault Server

    Note over App,Vault: 3. Token Renewal Process

    App->>Vault: POST /v1/auth/token/renew-self
    Note right of App: X-Vault-Token: <current_token>

    alt Token renewal successful
        Vault-->>App: new lease_duration
        Note left of Vault: Token TTL extended

        App->>App: token_issued = now
        App->>App: token_expiry = now + lease_duration
        Note over App: Update token information

    else Token renewal failed
        Note over App: Token renewal not possible
        App->>App: Log error and send notification
        App->>App: Terminate application or use default values
    end
```

### 🔑 1.3: Fetching Static Secrets (KV)

#### KV Secret Retrieval and Caching

```mermaid
sequenceDiagram
    participant Timer as Timer/Scheduler
    participant App as Application
    participant Cache as Local Cache
    participant Vault as Vault Server

    Note over Timer,Vault: 4. Fetching Static Secret (KV)

    App->>App: Secret request occurs

    alt Valid secret exists in cache
        App->>Cache: Retrieve cached secret
        Cache-->>App: Return cached secret
        Note over App: Use cached secret

    else Cache empty or expired
        App->>Vault: GET /v1/kv-demo/data/sample
        Note right of App: X-Vault-Token: <valid_token>

        Vault-->>App: Secret data + metadata
        Note left of Vault: data.data structure

        App->>Cache: Store secret in cache
        Note over App: Set cache TTL (based on config file)

        App->>App: Use secret data
    end

    Note over Timer: Periodic cache refresh (configured interval)
    Timer->>App: Request cache refresh
    App->>Vault: GET /v1/kv-demo/data/sample
    Vault-->>App: Latest secret data
    App->>Cache: Update cache
```

### ⏰ 1.4: Managing Dynamic Secrets (TTL-based)

#### TTL-based Secret Renewal

```mermaid
sequenceDiagram
    participant Timer as Timer/Scheduler
    participant App as Application
    participant Vault as Vault Server

    Note over Timer,Vault: 5. Managing Dynamic Secrets (TTL-based)

    App->>Vault: GET /v1/database/creds/myapp-role
    Note right of App: X-Vault-Token: <valid_token>

    Vault-->>App: Secret + TTL information
    Note left of Vault: username, password, lease_duration

    App->>App: Start tracking secret TTL
    Note over App: Schedule renewal based on lease_duration

    loop Periodic renewal based on TTL
        Timer->>App: Request renewal before TTL expires
        Note over Timer: Renew at 4/5 of TTL

        App->>Vault: GET /v1/database/creds/myapp-role
        Note right of App: Use the same token

        Vault-->>App: New secret + TTL
        Note left of Vault: New credentials

        App->>App: Update with new secret
        App->>App: Schedule new renewal based on new TTL
    end
```

### 🔄 1.5: Integrated Process (Full Flow)

#### Complete Vault Integration Process

```mermaid
sequenceDiagram
    participant Timer as Background Timer
    participant App as Application
    participant Cache as Local Cache
    participant Vault as Vault Server

    Note over Timer,Vault: 6. Integrated Vault Process

    %% Initialization Phase
    App->>App: Application starts
    App->>App: Load configuration file (config.ini)

    %% Authentication Phase
    App->>Vault: POST /v1/auth/approle/login
    Note right of App: role_id, secret_id

    Vault-->>App: client_token + lease_duration
    App->>App: Store token info and calculate TTL

    %% Start Background Token Management
    App->>Timer: Start token renewal thread

    %% Main Loop
    loop Application is running
        %% Secret Request
        App->>App: Secret request occurs

        alt Static Secret (KV)
            App->>Cache: Check cache

            alt Cache is valid
                Cache-->>App: Cached secret
            else Cache is invalid
                App->>Vault: GET /v1/kv-demo/data/sample
                Vault-->>App: Secret data
                App->>Cache: Store in cache
            end

        else Dynamic Secret (Database/SSH, etc.)
            App->>Vault: GET /v1/database/creds/myapp-role
            Vault-->>App: Secret + TTL
            App->>App: Schedule renewal based on TTL
        end

        App->>App: Use the secret

        %% Background Processes
        par Token Renewal
            Timer->>App: Check token state
            alt Renewal needed
                App->>Vault: POST /v1/auth/token/renew-self
                alt Renewal successful
                    Vault-->>App: Updated token info
                else Renewal failed
                    App->>App: Log error and terminate
                    Note over App: Cannot renew due to token renewal failure
                end
            end
        and Cache Refresh (Static Secret)
            Timer->>App: Request cache refresh
            App->>Vault: GET /v1/kv-demo/data/sample
            Vault-->>App: Latest secret
            App->>Cache: Update cache
        and Dynamic Secret Renewal
            Timer->>App: Renew based on TTL
            App->>Vault: GET /v1/database/creds/myapp-role
            Vault-->>App: New secret
        end
    end

    %% Shutdown Process
    App->>App: Application shuts down
    App->>Timer: Terminate background threads
    App->>Cache: Clear cache
```

## Option 2: Using Vault Proxy (When Token Management is Not Feasible)

The conditions for using Option 2 are as follows:
- Option 2 is used when the token management part of Option 1's process is not feasible.
  - For scripts, implementing token management with multi-threading can be difficult.
- It is used when managing elements required for Vault login (e.g., AppRole Secret_id, Password) is difficult.

The process is the same as Option 1, but with the login and token management parts excluded. The rest is identical to Option 1.

1.  Fetching Static Secrets (KV)
2.  Managing Dynamic Secrets (TTL-based)
3.  Integrated Process (Full Flow)

## Option 3: Vault Agent + Environment Variable/File Injection (For Legacy Apps)

The conditions for using Option 3 are as follows:
- Option 3 is used when the token management part of Option 1's process is not feasible.
- It is used for permanently running services rather than one-off tasks like scripts.
- It is used when managing elements required for Vault login (e.g., AppRole Secret_id, Password) is difficult.

### 🔧 3.1: Execute Command After Creating Environment Variables

#### Vault Agent Environment Variable Injection Process

```mermaid
sequenceDiagram
    participant Vault as Vault Server
    participant Agent as Vault Agent
    participant Env as Environment Variables
    participant App as Application

    Note over Vault,App: 1. Vault Agent Environment Variable Injection Process

    %% Agent Initialization
    Agent->>Agent: Start Vault Agent
    Agent->>Agent: Load configuration file (vault-agent.hcl)

    %% Vault Authentication
    Agent->>Vault: POST /v1/auth/approle/login
    Note right of Agent: Send role_id, secret_id

    Vault-->>Agent: client_token + lease_duration
    Note left of Agent: Agent handles token management

    %% Create Environment Variables
    Agent->>Vault: GET /v1/kv-demo/data/sample
    Note right of Agent: X-Vault-Token: <agent_token>

    Vault-->>Agent: Secret data
    Agent->>Env: Create environment variables
    Note over Env: DB_PASSWORD=secret123
    Note over Env: API_KEY=key456

    %% Execute Application
    Agent->>App: Execute with environment variables
    Note right of Agent: exec { command = ["./my-app"] }

    App->>App: Use getenv("DB_PASSWORD")
    App->>App: Use getenv("API_KEY")

    %% Periodic Renewal
    loop Detect secret changes
        Agent->>Vault: GET /v1/kv-demo/data/sample
        Vault-->>Agent: Latest secret data

        alt Secret has changed
            Agent->>Env: Update environment variables
            Agent->>App: Restart application
            Note over Agent: restart_on_secret_changes = "always"
        else Secret is the same
            Note over Agent: No change, continue running
        end
    end
```

## 🎯 Core Principles

### 1. **Token Management**
- Renew token at 4/5 of its TTL.
- Do not attempt to re-login on renewal failure (possibility of `secret_id` expiration).
- Continuously monitor in the background.

### 2. **Secret Caching**
- Static Secrets: Configuration-based cache TTL.
- Dynamic Secrets: Automatic renewal based on Vault TTL.
- Store only in memory, do not save to files.

### 3. **Error Recovery**
- On token renewal failure, terminate the application or use default values.
- Do not attempt to re-login if the `secret_id` expires.
- On failure, ensure a safe shutdown or use default values.

### 4. **Security Considerations**
- Use secrets only in memory.
- Do not print secret data in logs.
- Set appropriate token expiration times.
- Perform regular security audits.
